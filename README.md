## Overview

TorahAi is a link-in-bio backend API (like Linktree) built to demonstrate production-grade Go backend development. The project covers:

- Clean vertical slice architecture (`auth/`, `links/`, `profile/`)
- PostgreSQL with raw SQL queries via SQLC (no ORM)
- Database migrations with `golang-migrate`
- JWT authentication
- Structured error handling (Postgres errors + app errors)
- Dev and production environment separation
- Docker-based local development

## Prerequisites

Make sure you have the following installed before you begin:

- [Go 1.21+](https://go.dev/dl/)
- [Docker](https://www.docker.com/) and Docker Compose
- [golang-migrate CLI](https://github.com/golang-migrate/migrate)
- [sqlc CLI](https://docs.sqlc.dev/en/latest/overview/install.html)
- [air](https://github.com/air-verse/air) (for live reload)

### Install the CLI tools

```bash
# Install migrate with the postgres driver — this flag is required
go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest

# Verify migrate
migrate -version

# Install sqlc
go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest

# Verify sqlc
sqlc version

# Install air for live reload
go install github.com/air-verse/air@latest
```

## Getting Started

### 1. Clone the repository

```bash
git clone https://github.com/taiwoajasa245/TorahAi-api.git
cd TorahAi-api
```

### 2. Install Go dependencies

```bash
go mod tidy
```

### 3. Set up your environment file

Copy the example env file and fill in your values:

```bash
cp .env.production .env.development
```

Then open `.env.development` and update the values — see [Environment Variables](#environment-variables) for details.

### 4. Start the database

This spins up a PostgreSQL container using Docker Compose:

```bash
make docker-run
```

### 5. Run database migrations

This creates all the tables in your database:

```bash
make migrate-up
```

### 6. Start the server

For development with live reload:

```bash
make watch
```

Or run directly:

```bash
make run-dev
```

The API will be available at `http://localhost:8080`.

> **Important:** Never commit `.env.development` or `.env.production` to Git. Only `.env.example` should be committed — it shows teammates what keys are needed without exposing secrets.

For production, set environment variables directly on your server or via your deployment platform (Railway, Render, Fly.io etc). The app reads from OS environment variables when no `.env` file is found.

## Database & Migrations

This project uses [golang-migrate](https://github.com/golang-migrate/migrate) to manage database schema changes. Every change to the database schema must be done through a migration file — never edit the database manually.

### Creating a new migration

```bash
make migrate-create name=create_links
```

This generates two files inside `db/migrations/`:

```
db/migrations/
├── 000002_create_links.up.sql      # what to do (create table)
└── 000002_create_links.down.sql    # how to undo it (drop table)
```

Always write both the `.up.sql` and `.down.sql` files. The down file is your safety net for rolling back.

**Example migration:**

```sql
-- 000002_create_links.up.sql
CREATE TABLE IF NOT EXISTS links (
    id         SERIAL PRIMARY KEY,
    user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title      VARCHAR(255) NOT NULL,
    url        TEXT NOT NULL,
    position   INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 000002_create_links.down.sql
DROP TABLE IF EXISTS links;
```

### Running migrations

```bash
# Apply all pending migrations
make migrate-up

# Roll back the last migration
make migrate-down
```

> **Rule:** Every time you add a new feature that needs a new table or column, create a migration first, run it, then write your queries.

## Code Generation with SQLC

This project uses [sqlc](https://sqlc.dev) to generate type-safe Go code from raw SQL queries. You write SQL, sqlc writes the Go repository code for you.

### The workflow every time you add a new query

**Step 1 — Write your SQL query in `db/queries/`**

```sql
-- db/queries/links.sql

-- name: CreateLink :one
INSERT INTO links (user_id, title, url, position)
VALUES ($1, $2, $3, $4)
RETURNING *;

-- name: GetLinksByUserID :many
SELECT * FROM links
WHERE user_id = $1
ORDER BY position ASC;

-- name: DeleteLink :exec
DELETE FROM links
WHERE id = $1 AND user_id = $2;
```

**Step 2 — Generate the Go code**

```bash
make sqlc-generate
```

This reads your SQL queries and generates ready-to-use Go functions inside `db/sqlc/`. You never edit files in `db/sqlc/` manually — they are always regenerated.

**Step 3 — Use the generated code in your repository**

```go
// internal/links/repository.go
func (r *repository) CreateLink(ctx context.Context, req CreateLinkRequest) (*Link, error) {
    result, err := r.queries.CreateLink(ctx, db.CreateLinkParams{
        UserID:   req.UserID,
        Title:    req.Title,
        Url:      req.URL,
        Position: req.Position,
    })
    if err != nil {
        return nil, err
    }
    return fromDBLink(result), nil
}
```

> **Rule:** Never write raw SQL inside your Go files. All SQL lives in `db/queries/` and all DB calls go through the sqlc generated `db/sqlc/` package.

## Development Workflow

Here is the recommended order of steps every time you build a new feature:

### Step 1 — Create the migration

```bash
make migrate-create name=create_links
```

Write the SQL schema in the generated `.up.sql` and `.down.sql` files.

### Step 2 — Run the migration

```bash
make migrate-up
```

### Step 3 — Write the SQL queries

Add your queries to `db/queries/your_feature.sql` using sqlc annotations:

```sql
-- name: YourQueryName :one   (returns one row)
-- name: YourQueryName :many  (returns many rows)
-- name: YourQueryName :exec  (returns no rows)
```

### Step 4 — Generate the Go code

```bash
make sqlc-generate
```

### Step 5 — Build the feature slice

Create your feature folder inside `internal/` with these files:

```
internal/your_feature/
├── model.go        # structs: domain model, request, response
├── repository.go   # calls sqlc — only place that touches the DB
├── service.go      # business logic
└── handler.go      # HTTP handlers — reads request, calls service, writes response
```

### Step 6 — Register the routes

Add your routes to `internal/router/router.go`.

### Step 7 — Test it

```bash
make test       # unit tests
make itest      # integration tests
```

## API Endpoints

All responses follow this standard format:

**Success:**
```json
{
  "status": 200,
  "success": true,
  "message": "user registered successfully",
  "data": { ... }
}
```

**Error:**
```json
{
  "status": 409,
  "success": false,
  "message": "a record with this 'email' already exists"
}
```

### Auth

| Method | Endpoint | Description | Auth Required |
|---|---|---|---|
| `POST` | `/api/v1/auth/register` | Register a new user | No |
| `POST` | `/api/v1/auth/login` | Login and get JWT token | No |

### Links

| Method | Endpoint | Description | Auth Required |
|---|---|---|---|
| `GET` | `/api/v1/links` | Get all links for current user | Yes |
| `POST` | `/api/v1/links` | Create a new link | Yes |
| `PUT` | `/api/v1/links/:id` | Update a link | Yes |
| `DELETE` | `/api/v1/links/:id` | Delete a link | Yes |

### Profile

| Method | Endpoint | Description | Auth Required |
|---|---|---|---|
| `GET` | `/api/v1/profile` | Get current user profile | Yes |
| `PUT` | `/api/v1/profile` | Update current user profile | Yes |
| `GET` | `/api/v1/u/:slug` | Get public profile by slug | No |

> **Protected routes** require a `Bearer` token in the `Authorization` header:
> ```
> Authorization: Bearer <your_jwt_token>
> ```

## Error Handling

The API has a two-layer error handling system in `pkg/apperror/`:

**Layer 1 — App errors** — for non-database errors where you want a specific HTTP status:

```go
// in your service
return nil, apperror.New(http.StatusUnauthorized, "invalid email or password")
return nil, apperror.New(http.StatusNotFound, "user not found")
```

**Layer 2 — Postgres errors** — automatically detected and mapped to clean messages:

| Postgres Code | What It Means | HTTP Status Returned |
|---|---|---|
| `23505` | Unique constraint violated | `409 Conflict` |
| `23502` | Not null constraint violated | `400 Bad Request` |
| `23503` | Foreign key violation | `422 Unprocessable Entity` |
| `40P01` | Deadlock detected | `409 Conflict` |
| `53300` | Too many connections | `503 Service Unavailable` |
| `57014` | Query cancelled (timeout) | `408 Request Timeout` |

**In your handler — one line handles everything:**

```go
result, err := h.service.DoSomething(r.Context(), req)
if err != nil {
    apperror.Handle(w, err)   // handles app errors + all postgres errors
    return
}
```

## Project Architecture

The app follows a **vertical slice** architecture — each feature owns all its layers:

```
HTTP Request
    ↓
Router (net/http)       internal/router/router.go
    ↓
Middleware              internal/auth/middleware.go
    ↓
Handler                 internal/feature/handler.go     (decode request, encode response)
    ↓
Service                 internal/feature/service.go     (business logic, validation)
    ↓
Repository              internal/feature/repository.go  (sqlc calls only)
    ↓
SQLC Generated Code     db/sqlc/
    ↓
PostgreSQL
```

**Key rules:**

- `db.User` (sqlc type) never leaves `repository.go` — it is converted to your domain type immediately
- Services never import sqlc directly — they only talk to the repository interface
- Handlers never contain business logic — they only decode, call service, encode
- All SQL lives in `db/queries/` — never write raw SQL inside Go files
